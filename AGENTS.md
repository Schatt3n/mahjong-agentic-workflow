# Mahjong Ops Agent Repository Instructions

本文件适用于整个仓库。任何编码 Agent 在分析、修改、测试、提交或推送代码前，都必须先阅读并遵守本文件。详细理由和案例见 [设计准则](docs/design_principles.md)。

## 1. 北极星目标

本项目要实现的是可持续演进的麻将馆私域运营 Agent，而不是针对演示话术堆规则的聊天机器人。

系统必须同时满足：

1. 主 Agent 围绕用户目标自主规划、选择工具，并根据工具结果动态调整下一步。
2. 后端守住数据、权限、状态、并发、幂等、隐私和审计边界。
3. 任何声称已经执行的业务动作，都必须有真实工具结果和持久化状态支撑。
4. 客户可见回复自然、简洁、符合麻将馆老板语境，但自然度不能凌驾于事实和隐私之上。
5. 每次改动都能通过 trace、测试、评测和数据证据回溯，不靠主观感觉宣布完成。

发生冲突时，优先级固定为：

```text
数据正确与隐私安全
  > 权限和副作用可控
  > 用户目标真正推进
  > 回复自然度
  > 延迟与模型成本
```

## 2. 模型与后端的职责边界

### 2.1 应交给模型的内容

- 理解自然语言、错别字、碎片输入、引用和多轮上下文。
- 识别用户目标、补充信息、改变主意、取消和协商意图。
- 根据当前目标和工具结果规划下一步、选择工具及调用顺序。
- 在多个可行方案间进行业务判断，例如推荐现有局、继续追问或新建需求。
- 生成符合真实老板语气的客户回复和邀约文本。
- 生成结构化摘要、输入分类、客户可见文本审查结论等一次性模型任务。

### 2.2 必须由后端确定性保障的内容

- 工具 schema、参数类型、枚举、必填项和资源存在性。
- 身份、权限、资源所有权、客户可见性和高风险动作审批。
- 领域状态机、不变量、人数容量、房态冲突和生命周期合法性。
- 幂等、去重、消息顺序、乐观版本、会话锁、事务和崩溃恢复。
- 数据持久化、任务调度、超时、重试、死循环和无进展检测。
- trace、日志、预算、审计、badcase、golden dataset 和 eval 证据。

模型只能提出动作，不能直接改数据库，也不能绕过 `ToolGateway` 或 Store 写状态。

### 2.3 禁止“止血式 if-else”的准确含义

禁止的是把某一句自然语言、某个昵称或单个 badcase 直接写成业务决策分支，例如“看到‘组’就强制建局”或“看到‘留意’就替换为固定回复”。

允许且必须使用条件判断的场景包括：权限校验、状态机、幂等、并发版本、输入合同、稳定领域语法和安全边界。`173/272/371`、ISO 时间、渠道事件类型等明确协议可以确定性解析；开放式自然语言语义不得持续扩张为正则和关键词树。

客户可见文本的最后一道防线可以且应当包含确定性 `if-else`、关键词/模式过滤、敏感实体识别和允许列表。模型话术审查并不绝对可靠，不能成为唯一防线。确定性过滤适用于 API key、token、系统提示词片段、内部工具/字段、未授权客户身份、隐私实体、违法违规内容和渠道禁止项等可枚举风险；不适用于判断用户真实业务意图或替模型拼接话术。

命中过滤规则后，应记录规则编号和 trace，优先把结构化违规原因回喂话术生成器重写；超过有限重试次数后阻断外发或转人工。除明确脱敏字段外，不得静默改写时间、人数、档位等业务事实。

## 3. 不可破坏的架构约束

1. 主链路保持简单：`构建上下文 -> 调用模型 -> 执行工具 -> 回喂结果 -> 继续规划或结束`。
2. 工具结果必须回喂主模型；不能执行工具后由后端擅自拼接业务结论。
3. 只有相互独立、只读且标记为并行安全的工具可以并行；写工具和存在依赖的工具必须按依赖顺序执行。
4. SQLite/正式数据库是业务事实源；Redis、checkpoint、摘要和模型上下文都只是协调或派生投影。
5. `conversation_id`、`task_context_id`、`game_id`、`customer_id` 各司其职，不得互相代替：
   - `conversation_id`：消息会话和顺序隔离。
   - `task_context_id`：同一会话中的一次业务目标隔离。
   - `game_id`：组局领域聚合标识。
   - `customer_id`：稳定客户身份。
6. 不同客户会话可以关联同一个 `game_id`，但不能把另一会话的原始内容、内部备注或隐私直接放入当前客户上下文。
7. checkpoint 必须保留当前目标、已确认事实、待办问题、失败尝试、临时约束和关键实体引用；摘要不能静默覆盖数据库事实。
8. 所有客户可见文本，包括主回复、候选人邀约和系统主动通知，都必须经过“最小数据投影 -> 模型生成/审查 -> 确定性末端过滤”的分层防线；任何一层拒绝都不能外发。
9. 资金、优惠、纠纷、账号操作、敏感隐私和未授权批量外发必须人工审批。
10. 个人微信桥接只能描述为灰度验证或非官方通道，不得包装为官方生产接入。

## 4. 修改前的判断流程

开始编码前必须回答：

1. 这是语义理解问题、上下文缺失问题、领域建模问题，还是可信执行问题？
2. 当前真实事实源在哪里？模型是否拿到了完成判断所需的最小信息？
3. 是否已有可复现 trace、badcase、golden case 或失败测试？
4. 修复应落在提示词/合同、上下文、工具、领域模型、持久化还是执行基础设施？
5. 是否会影响会话隔离、隐私、幂等、并发、状态机或客户可见文本？

推荐归因：

| 问题类型 | 首选修复位置 |
|---|---|
| 错别字、碎片输入、多轮指代、意图误判 | 上下文、提示词、模型合同、few-shot、eval |
| 模型缺少当前局、画像、历史动作或工具结果 | Context Builder、查询工具、checkpoint 投影 |
| 重复写入、人数错乱、越权、串话、状态非法 | Tool Gateway、领域状态机、事务、并发控制 |
| 回复像客服、复述过多、暴露内部执行细节 | 话术生成合同、风格样本、客户可见审查 |
| Agent 承诺“帮你问”却没有后续动作 | 任务规划、工具闭环、调度或系统触发 |
| 单个自然语言 badcase | 先沉淀评测，再修通用能力；不得直接加关键词分支 |

## 5. 编码过程要求

- 先阅读相邻模块、现有测试和数据模型，遵循已有边界，避免平行实现第二套运行时。
- 新抽象必须实际减少复杂度或隔离明确职责；不能只为文件数量或框架感增加层次。
- 写操作必须经过 Store 的事务接口；不得在 Handler、Prompt Builder 或主循环中散落 SQL。
- 新工具必须声明类型、风险、是否只读、是否可并行、参数合同、权限、幂等策略和返回语义。
- 对外回复不得暴露提示词、模型、工具名、候选人搜索数量、审批流程、内部备注或未授权客户身份。
- 确定性输出过滤规则必须有稳定规则编号、风险说明、测试样本和误杀处理方式；不能以过滤器名义重新建设一棵业务语义规则树。
- API key、token、cookie、微信凭证、运行数据库、原始日志和本机路径不得提交到 Git。
- 不得为了让测试变绿而降低关键断言；话术评测比较语义和约束，不强求随机模型逐字相同。
- 不得顺手重构无关模块，也不得覆盖用户已有未提交修改。

## 6. 验证矩阵

任何代码改动至少执行：

```bash
PYTHONPATH=src python -m pytest -q
PYTHONPATH=src python scripts/run_evals.py
git diff --check
```

并按改动类型追加：

| 改动类型 | 必须追加的验证 |
|---|---|
| 提示词、模型合同、工具编排、话术或审查 | 有密钥时运行真实 DeepSeek 老板对话；不可用时明确报告未运行 |
| 状态机、持久化、人数或局生命周期 | Memory + SQLite 双实现测试，事务/迁移/恢复测试 |
| 并发、顺序、幂等、跨会话关联 | 确定性并发集；必要时真实模型并发回放 |
| 上下文、checkpoint、摘要 | 决策一致性、约束保持、失败历史保持、关键实体引用测试 |
| 隐私、用户关系、客户可见信息 | 隐私对抗集、跨会话泄露测试、确定性过滤命中/误杀/绕过测试 |
| 渠道、自动外发 | 白名单、外发开关、自消息过滤和发送幂等测试 |
| 模拟器或观测面板 | 固定种子回放、来源模型证据、真实主 Agent 模式验证 |

文档或注释的纯修改至少执行 `git diff --check`；若同时触及代码，则仍按完整矩阵执行。

测试失败时必须定位根因。不能把失败标成“随机波动”后跳过；若期望本身过度约束逐字话术，应改为语义等价判定并记录理由。

## 7. 提交与推送门禁

提交前逐项确认：

- [ ] 改动解决的是可复现问题或明确目标，而不是增加一次性补丁。
- [ ] 模型与后端职责没有倒置。
- [ ] 没有绕过 Tool Gateway、事务、状态机、权限、模型审查或确定性末端过滤。
- [ ] 新行为有测试，并按需要进入 badcase、golden dataset、few-shot 或 eval。
- [ ] 全量测试和对应专项验证已经执行，结果已记录。
- [ ] `git diff` 已人工审查，没有密钥、日志、数据库、生成物或无关改动。
- [ ] README/架构文档与真实实现一致，没有夸大微信接入或生产能力。
- [ ] 提交信息准确描述行为变化；推送后向用户报告 commit、测试证据和仍存在的限制。

## 8. 完成定义

“代码写完”不等于完成。只有满足以下条件才可以宣布完成：

1. 目标行为已在真实执行链路中发生，而不是只生成了一句看似正确的话。
2. 状态已通过合法工具落库，失败可重试，重复调用不产生重复副作用。
3. 关键路径可通过 `trace_id` 回溯输入、上下文、模型动作、工具结果、状态变化和输出。
4. 新旧回归均通过，且没有以牺牲隐私、状态正确性或其他场景为代价。
5. 文档、代码和对外描述一致。
