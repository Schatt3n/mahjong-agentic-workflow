# Mahjong Ops Agent 设计准则

> 状态：规范性文档。本文定义项目长期演进时的判断标准；根目录 [AGENTS.md](../AGENTS.md) 将其转换为编码 Agent 必须执行的仓库指令。

## 1. 我们究竟在做什么

本项目不是把 DeepSeek 接到聊天窗口，也不是把麻将术语写成一棵越来越大的规则树。目标是让一个 Agent 接受客户消息后，围绕“查局、组局、协商、邀约、确认、归档”等业务目标自主决定下一步，同时让后端保证每个动作可控、可信、可恢复、可审计。

因此系统采用“模型负责认知和规划，后端负责可信执行”的基本结构：

```text
渠道消息
  -> 输入聚合与任务分段
  -> 有界上下文
  -> 主 Agent 规划
  -> Tool Gateway 校验和执行
  -> 工具结果回喂
  -> 主 Agent 继续规划或回复
  -> 客户可见内容审查
  -> 通道发送
```

主 Agent 是唯一拥有业务目标推进权的长期循环。输入分类、摘要、话术生成、内容审查和模拟角色生成都是一次性模型任务，它们不能各自演化为另一个相互争夺状态的 Agent。

## 2. 设计决策的优先级

当正确性、体验、速度和成本冲突时，按以下顺序取舍：

1. **数据正确与隐私安全**：不能错加人、串会话、泄露内部备注或篡改局状态。
2. **副作用可控**：未经权限、状态和幂等校验，不能落库或外发。
3. **目标真正推进**：回复“我帮你问问”之后必须存在真实的候选人搜索、邀约草稿或调度任务。
4. **自然沟通**：像老板说话，少复述、少客服腔，但不能为自然度编造事实。
5. **延迟与成本**：在前四项满足后，再通过并行只读工具、上下文裁剪和模型分层优化。

这意味着“更快地给出错误答案”不是优化，“为了不转人工而擅自行动”也不是自动化。

## 3. 准则一：目标驱动，而不是规则驱动

主 Agent 每轮都应知道：当前用户目标是什么、已经完成了什么、还缺什么、有哪些工具、上一步工具返回了什么、现在能否安全结束。

合理的循环是：

```text
Goal -> Plan -> Act -> Observe -> Re-plan -> Stop
```

而不是：

```text
if 包含“组” -> 建局
elif 包含“有人吗” -> 固定回复
elif 包含“可以” -> 加入最近一个局
```

后者无法处理“可以倒是可以，但我最快四点半”“烟都可以，时间人齐再商量”“川麻那个我可以”等依赖上下文和工具结果的表达。

### 实施要求

- 主模型输出结构化 Action Contract，而不是让后端从自由文本猜动作。
- `objective_status=needs_tool` 时必须提供工具调用，且不能同时给客户一个假完成回复。
- 工具结果进入下一轮上下文，模型据此继续决策。
- Agent 只有在等待用户、目标完成、需要人工或明确失败时才能结束。
- Progress Monitor 检测重复工具调用、状态指纹不变和无进展循环。

## 4. 准则二：模型提出动作，后端决定动作是否合法

LLM 可以建议 `create_game`、`join_game`、`search_customers` 或 `send_message`，但不能直接写表。Tool Gateway 是模型世界与真实业务世界之间的信任边界。

每个副作用动作至少经过：

1. JSON/schema 合同校验。
2. 调用者身份、角色和资源所有权校验。
3. 领域状态机与业务不变量校验。
4. 权威幂等键、版本和并发冲突校验。
5. 高风险等级与人工审批校验。
6. 事务落库与 trace 记录。

模型传来的 `trace_id`、`idempotency_key` 或身份字段都只能视为建议；权威值由后端请求上下文生成或校验。

### 为什么这不是“Guard 捣乱”

合理的 Guard 不判断开放式业务语义，只阻止不合法动作。例如：

- “用户说‘组’，是不是意味着立即建局”属于模型判断。
- “同一客户能否同时确认到两个互斥局”属于领域不变量。
- “这次 join_game 是否重复提交”属于幂等。
- “当前会话能否看到另一客户的备注”属于权限和隐私。

## 5. 准则三：不为 badcase 无限增加语义分支

项目曾经出现过一种错误演进方式：每发现一句模型没理解的话，就在后端添加关键词、正则或固定回复。短期看修了一条，长期会造成规则互相覆盖、上下文失真和无法解释的行为。

每个 badcase 应先归因：

| 根因 | 正确处理 |
|---|---|
| 模型没看到相关事实 | 修 Context Builder 或查询工具 |
| 模型看到了但理解错误 | 修提示词、合同、few-shot、模型选型或评测样本 |
| 数据结构表达不了事实 | 修领域模型和迁移 |
| 模型提出了非法动作 | Tool Gateway 返回结构化错误，回喂模型重规划 |
| 动作正确但没有继续执行 | 修 Agent Loop、任务调度或系统触发 |
| 话术不自然 | 修话术生成合同和真实老板样本 |

### 允许确定性解析的边界

确定、稳定、可枚举的协议可以使用代码解析：

- `173/272/371` 人数短码。
- `2-16/1-32` 等已经定义的档位结构。
- ISO 时间、UUID、渠道消息类型、数据库状态枚举。
- Schema、权限、幂等和状态机。

开放式自然语言、反讽、引用、多轮省略和关系协商应交给模型，不应靠关键词树模拟理解。

## 6. 准则四：领域事实只能有一个权威来源

SQLite/未来正式数据库保存权威业务事实。模型输出、上下文、Redis、摘要、页面缓存和日志都不是事实源。

关键标识必须分开：

| 标识 | 含义 | 不能替代什么 |
|---|---|---|
| `customer_id` | 稳定客户身份 | 不能用昵称替代 |
| `conversation_id` | 一个通道会话的顺序与隐私边界 | 不能代表一次组局任务 |
| `task_context_id` | 同一会话内的一次业务目标 | 不能代表客户身份 |
| `game_id` | 组局领域聚合 | 不能等同于发起人的会话 |
| `source_message_id` | 渠道原始消息幂等标识 | 不能代表语义重复 |
| `trace_id` | 一次处理链路的观测标识 | 不能作为业务幂等键本身 |

A、B、C 可以在各自隔离的会话里参与同一个 `game_id`。Store 只向每个会话投影其有权看到的信息，不能把 B 的原始消息或老板给 B 的内部备注暴露给 A。

### 聚合与状态

`Game` 是组局领域的核心聚合之一，但系统还存在 Customer、Conversation/Task Context、Waiting Demand、Outbound Draft 和 Scheduled Task 等独立生命周期对象。不能为了追求“一个聚合根”把所有数据重新塞回 Game JSON。

## 7. 准则五：上下文是有界投影，不是聊天记录全量复制

每次模型调用前，Context Builder 根据当前任务组装最小充分信息：

- 当前消息与必要的引用关系。
- 当前 `task_context_id` 的最近对话。
- 结构化 checkpoint。
- 发送者画像和本轮允许使用的关系事实。
- 与当前目标相关的 active games、等待需求和外发草稿。
- 可用工具合同。
- 上一轮工具结果。
- 输出合同和预算信息。

上午已经结束或失败的组局，不应因为还在同一个微信会话里而污染下午的新目标；如果用户明确说“还是上午那个局”，系统再通过引用、任务检索或实体标识召回旧任务。

### checkpoint 应保留什么

- `current_objective`：正在完成的目标。
- `confirmed_facts`：带来源的已确认事实。
- `pending_questions`：真正阻塞目标的问题。
- `failed_attempts`：已尝试但失败的搜索/邀约，防止压缩后重做。
- `completed_steps`：已经成功的工具动作。
- `temporary_constraints`：只对本次任务生效的限制。
- `entity_refs`：`game_id`、候选人状态、等待需求等关键引用。

摘要是派生数据。摘要与数据库冲突时以数据库为准；摘要质量通过“压缩前后决策一致性”验证，而不只验证 JSON 能否生成。

## 8. 准则六：说已经做了，就必须真的做了

以下回复都隐含真实副作用：

- “我帮你问问。”
- “加你 371 了。”
- “有消息告诉你。”
- “我记上了，明天帮你约。”

系统不能只生成这些话。相应链路必须存在候选人搜索/邀约草稿、参与者状态写入、等待需求登记或未来调度任务。若动作尚未执行，回复就不能使用完成时态。

这也是工具结果必须回喂模型的原因：模型需要知道动作究竟成功、失败、去重、被拒绝还是等待审批，才能形成真实回复。

## 9. 准则七：客户可见内容与内部执行信息分离

客户需要知道的是“有没有合适的局、时间、档位、烟况、当前缺口、接下来要不要确认”，而不是：

- 使用了哪个模型或工具。
- 搜索了多少名候选人。
- 草稿等待老板审批。
- 内部 `game_id`、trace、置信度或状态枚举。
- 老板对其他客户的备注、画像和私聊原文。

系统把业务决策、话术生成和隐私审查拆开：主 Agent 决定要传达的事实；话术任务把事实写成人话；审查任务只判断泄露与合规，不重新接管业务规划。

但模型审查仍然是概率系统，不能承担最后一道确定性安全边界。完整外发链路采用分层防御：

```text
最小权限的数据投影
  -> 模型生成自然话术
  -> 独立模型审查泄露与合规
  -> 确定性关键词/模式/敏感实体过滤
  -> 通道权限、白名单与发送幂等
  -> 真正外发
```

### 为什么末端允许使用 if-else

“不能堆 if-else”针对的是开放式业务语义，不针对确定性安全控制。即使生成模型和审查模型都返回通过，最终文本仍可以用代码检查：

- API key、token、cookie、身份证号、手机号等敏感模式。
- `system_prompt`、内部工具名、数据库字段、trace 或审批状态等内部实现信息。
- 未经授权的其他客户姓名、老板内部备注和私聊原文。
- 明确违法违规词、平台禁止项、恶意链接或脚本片段。
- 当前渠道、租户或客户配置的禁止外发词。

这些规则必须是稳定、可解释、可测试的风险边界，而不是“看到某句话就决定建什么局”的业务规则。关键词黑名单也不是万能的：它可能被同义改写绕过，也可能误伤正常表达，所以仍需最小数据投影、模型审查、实体识别和对抗测试共同工作。

### 命中末端过滤后的处理

1. 不直接发送。
2. 记录 `trace_id`、规则编号、命中类别和脱敏后的证据。
3. 将结构化违规原因回喂话术生成器，请求在不改变业务事实的前提下重写。
4. 达到有限重试上限仍不通过时，阻断外发或进入人工队列。
5. 只有手机号、证件号等定义清楚的字段可以按规则脱敏；不得静默修改时间、人数、档位、烟况和承诺状态。

因此，最终客户回复的安全不是“相信某一个审查模型”，而是依赖多层彼此独立的防线。

### 话术原则

- 默认不称呼客户姓名，除非画像明确记录了老板通常如何称呼。
- 信息齐全时简短确认，不复述所有槽位。
- 邀约只给对方做决定所需的信息。
- 不使用“耐心等待”“是否加入”“方便来吗”等客服腔模板。
- 不确定时承认不确定并追问，不能补造事实。

## 10. 准则八：可靠性来自幂等、顺序、事务和恢复

同一 `conversation_id` 默认串行处理，不同会话才并行。一个用户连续多条碎片消息先经过 Pending Input Buffer 聚合；新消息到来时，可取消或替换尚未外发的旧版本结果。

副作用工具必须具备：

- 后端生成的权威幂等键。
- 数据库唯一约束或幂等记录。
- 明确的状态迁移前置条件。
- 乐观版本或协调锁。
- 可恢复 lease/任务状态。
- 超时、有限重试和退避。
- 重复状态/无进展检测。

只读且独立的搜索可以并行。创建局后再搜索候选人、候选人确认后再更新局缺口等存在因果依赖的动作不能为了速度并行。

## 11. 准则九：主动任务也要回到同一条可信链路

未来预约、等待列表反向匹配、超时归档和候选人回复都可能由系统事件触发。它们不能绕过 Agent Runtime 直接拼一条消息外发。

统一模式是：

```text
持久化任务/领域事件
  -> Scheduler 或 Hook
  -> SystemTriggerMessage
  -> 目标会话协调锁
  -> 正常 Agent Loop
  -> 客户可见审查
  -> 幂等外发
```

这样主动消息仍然具备上下文、权限、trace、重试和审计能力。

## 12. 准则十：先可观测，再谈自动化

每次处理必须能按 `trace_id` 还原：

1. 原始输入和渠道元数据。
2. 输入聚合和分流决定。
3. Context Builder 实际投影内容及预算。
4. 模型请求、结构化输出、耗时和错误。
5. 工具参数、校验结果、幂等与状态迁移。
6. 客户可见生成和审查结果。
7. 最终回复、外发状态和后续调度任务。

日志要可审计但不能成为新的隐私泄露源。API key、cookie、完整敏感画像和不必要的私聊正文应脱敏或限制访问。

自动外发按风险逐步开放：先只生成草稿，再白名单自动发，最后才考虑更大范围；纠纷、资金、优惠和账号操作始终保留人工边界。

## 13. 准则十一：评测驱动演进，而不是演示驱动演进

每次发现问题，应形成以下闭环：

```text
真实 trace/badcase
  -> 最小可复现测试
  -> 根因分类
  -> 通用机制修复
  -> 聚焦测试
  -> 全量 pytest
  -> 统一 eval
  -> 必要时真实模型回放
  -> 更新文档和基线
```

评测至少覆盖：

- 意图与槽位语义正确性。
- 工具选择和参数正确性。
- 状态推进与副作用一致性。
- 多轮、碎片、取消、改口和任务切段。
- checkpoint 压缩前后决策一致性。
- 跨会话隐私和对抗样本。
- 并发、幂等、重复消息、失序和恢复。
- 客户话术的语义约束与老板风格。
- P50/P95/P99、模型调用次数、token 和失败率。

随机生成式话术不应按逐字相等判失败。评测应判断关键事实、承诺、隐私和业务动作是否等价。

## 14. 准则十二：架构复杂度必须有业务回报

主循环本身应保持短小。复杂度可以存在于明确边界中，例如 Store、Tool Gateway、Context Lifecycle、Scheduler 和可观测性，但每层必须有独立职责和测试。

新增框架、多 Agent、消息队列、向量库或分布式锁之前，要回答：

1. 当前真实问题是什么？
2. 现有边界为什么解决不了？
3. 新组件降低了什么复杂度，增加了什么运维成本？
4. 如何迁移、回滚和验证语义不变？

单店、几百客户、本地 MacBook 的目标下，SQLite + 单主 Agent 可以是合理选择。多节点时再引入共享数据库、共享协调器和按会话分区的消息队列；不能提前把所有互联网中台组件搬进来，也不能把本地方案夸大为分布式生产架构。

## 15. 一次改动应该怎样落地

以“用户说‘可以’，系统没有继续找候选人”为例：

### 错误做法

在 Handler 中增加：

```python
if text == "可以":
    search_customers()
```

它不知道“可以”是在同意什么，也绕过了主 Agent、工具合同和任务状态。

### 正确做法

1. 用 trace 还原上一轮 Agent 的真实回复是否进入 recent conversation。
2. 检查当前 `task_context_id` 和 checkpoint 是否保留“等待用户确认组局”。
3. 检查模型合同能否表达“确认后继续调用工具”。
4. 检查 `search_customers` 是否在当前状态可用，结果是否回喂。
5. 将该多轮对话加入 golden/eval。
6. 修复上下文、合同或工具闭环中的通用缺口。
7. 跑同类“行/组/可以/那就问问”语义变体，而不是只测原句。

## 16. 提交前完成定义

一次变更只有同时满足以下条件，才能进入主分支：

- 行为在完整链路发生，不是页面或回复文本假象。
- 数据状态、客户回复和工具结果一致。
- 失败可重试，重复调用无重复副作用。
- 会话、任务、客户和局的标识没有混用。
- 客户可见信息同时通过模型审查和确定性末端过滤。
- 新问题进入测试或评测资产。
- 全量 `pytest`、统一 `run_evals.py` 和对应专项验证通过。
- diff 不含密钥、运行日志、数据库、缓存和无关重构。
- README、架构文档、测试数据和真实能力一致。
- 提交后报告 commit、测试证据、真实模型是否运行及仍存限制。

## 17. 代码审查时的十个问题

1. 这次是在增强通用能力，还是只修一句话？
2. 这个判断应该由模型、后端还是人工做？
3. 模型看到的信息是否足够且最小？
4. 是否存在直接写数据库或绕过 Tool Gateway 的路径？
5. 工具成功、失败、去重、超时后，模型分别会看到什么？
6. 同一消息重放、同一工具重试、两个会话并发时会怎样？
7. 会不会把其他客户的原文、备注或关系信息暴露出去？模型审查之后，确定性末端过滤还能拦住哪些风险？
