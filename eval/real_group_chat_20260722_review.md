# 2026-07-22 真实群聊数据清洗记录

## 采集范围

- 通道：WeChaty 只读观察链路。
- 观察对象：3 个已配置的麻将组局群，提交数据中统一匿名为 `room_alpha`、`room_beta`、`room_gamma`。
- 截止时间：2026-07-22 22:00（Asia/Shanghai）。
- 原始观测：112 条，其中 70 条包含文本。
- 当前分类器判为业务消息：58 条。该标签仅用于定位候选消息，不作为评测真值。

## 清洗结果

| 分层 | 数量 | 用途 |
| --- | ---: | --- |
| Gold | 12 | 业务含义和期望动作明确，可进入自动回归 |
| Adversarial | 4 | 指代、地方黑话或上下文缺失，保留候选结论和待确认问题 |

Gold 覆盖以下生产行为：

1. 长模板和紧凑格式的看板字段提取。
2. 编号看板整体替换与单条增量追加。
3. 同一局由 `272` 到 `371` 的人数进展，禁止重复建局。
4. 群成员查询当前局，查询动作不得创建或加入局。
5. 同人连续消息聚合以及档位条件放宽。
6. `三缺一` 的人数语义和缺失槽位保护。
7. 引用看板后回复“人齐”的目标定位与状态推进。
8. 表情、置顶、拍一拍等确定性噪声零模型调用。
9. 撤回消息失效，禁止使用已撤回文本推进状态。

Adversarial 保留以下真实难题：

1. 客户代表两个座位，同时接受“没有两个就一个”的降级方案。
2. “红中晚上 3 块”是否能稳定映射到本地档位码 `368`。
3. “那个一打不了”的指代对象和拒绝原因不明确。
4. 群成员自行发布“川麻换三 132 371”后，老板的手势回复是否代表已纳入看板。

## 真实运营观察

- 老板绝大多数消息用于发布或更新缺人看板，符合“群聊是公告板”的产品假设。
- 群内仍会出现少量短对话，例如确认客户需要一个还是两个座位。因此不能把“老板绝不在群里对话”写成硬规则。
- 多轮短句必须同时包含客户和老板的消息。只按发送者或逐条分类，会错误理解“可以两个”“没有的话就一个”。
- 看板快照是群聊 Session 的公共状态；单个 Session 只应包含与本话题相关的消息，不能把整个群历史原文喂给模型。

## 匿名化规则

- 不提交真实群名、昵称、微信 ID、头像、签名和完整通道 payload。
- 房间和参与者只使用 `room_*`、`operator_*`、`customer_*` 别名。
- 原始消息 ID 仅保留截断 SHA-256，用于本地审计时核对来源。
- 不复制当前模型输出的 `semantic_action`、置信度或业务分类作为期望答案。
- 含歧义的样本不得为了让当前实现通过而强行标成 Gold。
